프로젝트

GXemul 환경에서 커널 부팅부터 가상 메모리, 프로세스/시스템콜, pipe IPC, 파일시스템까지 단계적으로 구현했습니다. 특히 TLB Fast Refill, Copy-on-Write 기반 fork(), 사용자 공간 FS 서버와 IPC를 통한 파일시스템 구조를 구현하며 운영체제 핵심 메커니즘을 실습했습니다.

주요 개발 내용

  • 가상 메모리 체계 구축: 2단계 페이지 테이블 기반 가상 메모리와 TLB Fast Refill 메커니즘 구현.
  • 프로세스 최적화: fork() 시스템 콜에 Copy-on-Write(COW) 방식을 적용하여 메모리 복사 비용 최적화.
  • IPC 메커니즘: pipe IPC 및 파일 디스크립터 복제/참조 관리 로직 구현.
  • 마이크로커널 아키텍처: 커널 내부 파일시스템 대신 사용자 공간 FS 서버와 IPC로 통신하는 구조 설계.

트러블슈팅

Pipe IPC 구현 중 매핑 순서에 따른 참조 수(pageref) 판정 오류 해결
Kernel / IPC
상황

dup() 및 close() 실행 시, FD와 Pipe 페이지의 매핑·해제 순서에 따라 pageref 값이 일시적으로 의도와 다르게 관측되는 레이스 컨디션이 발생했습니다.

대안 탐색

인터럽트 비활성화를 통한 임계 구간 보호를 검토했으나, 시스템 성능 저하와 교육용 OS의 설계 원칙을 고려해 제외했습니다. 대신 매핑 순서 재배치(Reordering)를 통해 항상 안전한 상태 불변식을 유지하는 방안을 선택했습니다.

해결

pageref(fd) ≤ pageref(pipe) 조건이 어떤 순간에도 깨지지 않도록 unmap·map 순서를 조정했습니다. IPC의 안정성을 확보했습니다.

pipe.c — Atomic-like Mapping Sequence
// 해제 순서 조정으로 불변식 유지
void pipe_close(int fdnum) {
  // [Fix] FD를 먼저 해제하여
  // pageref(fd) > pageref(pipe)
  //   찰나의 순간을 방지
  syscall_mem_unmap(0, fd_to_va(fdnum));
  syscall_mem_unmap(0, pipe_to_va(fdnum));
}
💡 알게된 점

선점형 스케줄링 환경에서는 '완성된 결과'뿐만 아니라 '과정 중의 중간 상태'도 안전해야 함을 깨달았습니다.

사용 기술

CMIPS AssemblyMIPS SimulatorMicrokernelIPC (Pipe)

SysY 언어의 렉서(Lexer), 재귀 하향 파서(Recursive Descent Parser), P-Code 생성기, 스택 머신 기반 가상 머신 인터프리터를 바닥부터 직접 설계·구현했습니다.

주요 개발 내용

  • 정교한 에러 처리: 파싱 중 세미콜론·괄호 누락, 미선언 변수, 상수 재대입 등 다양한 에러 유형을 판별하는 에러 핸들링 로직 설계.
  • 계층적 심볼 테이블: 중첩 스코프를 올바르게 관리하는 계층적 심볼 테이블 설계 및 타입·상수 검증 로직 구현.
  • P-Code 가상 머신 구현: 스택 기반 VM을 직접 설계하여 배열 인덱싱, 함수 호출/반환, I/O를 완벽 지원.

트러블슈팅

배열 파라미터 전달 시 P-Code 스택 주소 오류
Compiler / VM
상황

함수에 배열을 인자로 전달하면, 값이 아닌 base address가 복사되어야 하지만, 단순 LOD/STO를 사용하면 주소 대신 값이 전달되어 런타임 오류가 발생했습니다.

해결

배열 파라미터인지 여부를 심볼 테이블에 기록하고, 배열 인자일 경우 LOD(base_addr) → STO 대신 LDA(base_addr) → STO로 주소를 직접 전달하도록 코드 생성 단계를 수정했습니다.

codegen.cpp — Array Parameter Fix
// 배열 파라미터: 주소 전달
if (sym.isArrayParam) {
  emit(LDA, sym.level, sym.addr);
} else {
  emit(LOD, sym.level, sym.addr);
}
💡 알게된 점

컴파일러에서는 '값(value)'과 '주소(address)'의 구분이 코드 생성의 핵심이며, 심볼 테이블에 충분한 메타정보를 기록해야 올바른 코드 생성이 가능하다는 점을 체감했습니다.

주요 개발 내용

  • 설계 및 문서화 체계 구축: docs/ 및 RFC/를 중심으로 요구사항, 아키텍처, 데이터 설계, API 인터페이스 등 개발 전 과정의 문서를 명문화.

  • 공통 API 응답 규약 설계: ResponseResultHandler를 통한 응답 자동 래핑(code/message/data) 및 BusinessException 상속 체계 기반의 전역 예외 처리 로직 구현.
  • 프론트엔드 연동 및 통합: Vue 3 + TypeScript + Vite 환경과의 API 통신 구조 설계 및 프론트-백엔드 인터페이스 통합 수행.

트러블슈팅

백엔드 실행 환경 파편화 및 API 응답 규약 불일치 해결
백엔드 / 협업
상황

팀원 간 로컬 환경 설정(.env, AI 모델 경로) 차이로 인한 인프라 기동 실패가 발생했습니다. 또한, 개발자별로 상이한 API 응답 구조로 인해 프론트엔드에서 데이터 파싱 오류가 빈번하게 나타났습니다.

해결

README 기반 기동 프로세스를 확립하고, ResponseBodyAdvice를 활용해 모든 응답을 code/message/data 구조로 자동 래핑하는 아키텍처를 적용했습니다.

ResponseResultHandler.java
@RestControllerAdvice
public class ResponseResultHandler
    implements ResponseBodyAdvice<Object> {

  @Override
  public Object beforeBodyWrite(Object body, ...) {
    // 이미 공통 래퍼면 그대로 반환
    if (body instanceof ResultWrapper)
      return body;
    // 데이터를 규약에 맞춰 자동 래핑
    return ResultWrapper.success(body);
  }
}

협업 시 발생할 수 있는 휴먼 에러를 줄이기 위해서는 구두 약속보다 시스템(아키텍처)적으로 규약을 강제하는 설계가 팀 생산성에 기여한다는 점을 체감했습니다.

주요 개발 내용

  • 3-Tier 분리형 아키텍처 구축: 비즈니스 로직(Spring Boot)과 고부하 AI 연산(FastAPI)의 책임을 물리적으로 분리하여 운영 안정성을 확보했습니다.

시스템 성능 최적화 및 평가 지표

  • 실시간 AI 추론 속도: 12-lead ECG 데이터(10초 분량, 500Hz) 기준 AI 추론 속도는 평균 2382.85ms, p95 2698.16ms.
  • 모바일 체감 로딩 개선: Cache-First 적용 후 Home 화면 첫 표시 평균 0.29ms, 네트워크 동기화 완료 평균 604.3ms.

트러블슈팅

AI 모델 추론 지연 및 저사양 EC2 배포 중 OOM 문제
AI / DevOps
① 상황 — 실시간 추론 지연

초기 구조에서는 추론 지연과 모바일 환경의 자원 부담이 커서, 실시간 모니터링 서비스에 바로 적용하기 어려운 문제가 있었습니다.

① 해결

모델 업데이트 유연성과 모바일 메모리 부담 완화를 위해 서버 사이드 추론 방식을 채택하고, 모바일은 백엔드 프록시를 통해 결과만 전달받도록 구조를 분리했습니다.

② 상황 — 저메모리 EC2 OOM

AWS 저사양 인스턴스에서 컨테이너 빌드 또는 AI 서비스 동시 구동 시 메모리 사용량이 급증하면서 운영 안정성이 떨어지는 문제가 있었습니다.

② 해결 — Pull-only 배포

로컬 또는 CI 환경에서 이미지를 사전 빌드해 레지스트리에 Push하고, 운영 서버는 Pull-only 전략으로 전환했습니다. mem_limit 기반 자원 제한과 AI 컨테이너의 profile 분리를 적용했습니다.

docker-compose.ec2-lite.yml
services:
  backend:
    image: ghcr.io/zzooonn/carelink/backend:latest
    # build 섹션 제거 (EC2 내 빌드 방지)
    mem_limit: 384m

  ai:
    image: ghcr.io/zzooonn/carelink/ai:latest
    profiles: ["ai"]
    # 필요 시에만 분리 실행
    mem_limit: 512m
💡 알게된 점

제한된 인프라 자원에서는 단순 성능 튜닝보다, 아키텍처 분리·배포 전략·자원 제한을 함께 설계하는 접근이 전체 시스템 안정성에 더 직접적으로 기여합니다.
