트러블슈팅

Pipe 통신 구현 중 Context Switch로 인한 Race Condition 해결

Kernel / IPC
상황

dup()으로 파이프 FD를 복제하는 도중 타이머 인터럽트로 Context Switch가 발생하여 pageref 갱신이 꼬이면서 자식 프로세스가 쓰기 단이 닫힌 것으로 오인해 강제 종료되는 버그가 있었습니다.

대안 탐색

인터럽트 비활성화(cli/sti)로 임계 구간을 보호하는 방법과 매핑 순서 재배치 두 가지를 검토했습니다. 인터럽트 비활성화는 교육용 OS 설계 원칙상 금지되어 있어, 매핑 해제·연결 순서를 변경해 항상 안전한 불변식을 유지하는 방향을 선택했습니다.

해결

pageref(fd) ≤ pageref(pipe) 조건이 항상 유지되도록 unmap·map 순서를 재배치(Reordering)하여 원자성 부족 문제를 우회하고 안전한 IPC 환경을 구축했습니다.

pipe.c — Mapping Sequence Fix 
// fd와 pipe 해제 순서가 핵심인 close 로직
void pipe_close(int fdnum) {
  // [수정 후] fd를 먼저 해제 →
  // pageref(fd) ≤ pageref(pipe) 유지
  syscall_mem_unmap(0, fd_to_va(fdnum));
  syscall_mem_unmap(0, pipe_to_va(fdnum));
}
알게된 점

하드웨어 타이머로 인해 코드는 언제든 멈출 수 있음을 체감했습니다. 선점형 스케줄링 환경에서는 동시성과 실행 순서를 항상 의심하고 방어적으로 설계해야 한다는 기본기를 다졌습니다.

트러블슈팅

파서의 Fault-Tolerant 파싱 및 에러 복구 설계

Compiler
상황

세미콜론·괄호 누락 시 컴파일러가 강제 종료(Panic)되거나, 렉서가 이미 다음 줄 토큰을 읽어 엉뚱한 위치에 구문 오류를 보고하는 고질적 이슈가 발생했습니다.

대안 탐색

에러 발생 즉시 파싱을 중단하는 방식과, 에러를 기록하고 파싱을 이어가는 회복(Recovery) 방식을 검토했습니다. 중단 방식은 첫 번째 오류 이후 모든 후속 오류를 놓치는 문제가 있어, 직전 토큰 위치를 추적해 복수 오류를 한 번에 보고하는 Recovery 방식을 선택했습니다.

해결

에러 복구(Error Recovery) 메커니즘을 설계했습니다. getPreviousToken()으로 직전 토큰 위치를 추적하여 정확한 에러 라인을 기록한 뒤 파닉 없이 파싱을 계속 이어나가도록 처리했습니다.

Parser.cpp — Error Recovery 
void parseErrorPrintfI() {
  Token cur  = getCurrentToken();
  Token prev = getPreviousToken();

  if (cur.type == SEMICN) {
    outputToken(cur);
    advanceToken();
  } else {
    // 누락(i) 에러 — 직전 토큰 라인 기록
    // 파닉 없이 파싱 계속
    addError(prev.lineNumber, 'i');
  }
}
알게된 점

에러 처리는 예외를 던지는 게 아니라, 잘못된 입력 속에서도 문맥을 잃지 않고 정상 궤도로 복귀하도록 설계하는 것이 견고한 엔지니어링의 핵심임을 깨달았습니다.

트러블슈팅

백엔드 인프라 파편화 및 API 공통 규약 불일치 해결

백엔드 / 협업
상황

팀원 간 로컬 설정(.env, AI 모델 경로)이 엇갈려 인프라 기동 실패가 잦았습니다. 여기에 API 응답 구조까지 제각각이어서 프론트 연동에 심각한 병목이 발생했습니다.

해결

README 기준으로 기동 환경을 통일하고, Controller는 비즈니스 데이터만 반환하면 ResponseResultHandler가 code/message/data 형태로 자동 래핑하도록 아키텍처를 개선했습니다. BusinessException 규약도 강제하여 혼선을 제거했습니다.

ResponseResultHandler.java 
@RestControllerAdvice
public class ResponseResultHandler
    implements ResponseBodyAdvice<Object> {

  @Override
  public Object beforeBodyWrite(Object body, ...) {
    // 이미 공통 래퍼면 그대로 반환
    if (body instanceof ResultWrapper)
      return body;
    // 데이터를 code/message/data로 래핑
    return ResultWrapper.success(body);
  }
}

다중 스택 협업에서는 개인 역량보다 인터페이스 규약을 시스템적으로 강제하는 아키텍처 설계가 팀 전체 생산성에 직결됨을 배웠습니다.

① 상황 — 대용량 모델 실시간 지연

초기 1D-ResNet 적용 시 모바일에서 평균 추론 지연 2,300ms(p95: 2,395ms), 배터리 소모 문제가 발생했습니다.
