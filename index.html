GXemul 환경에서 Lab0~Lab6의 7단계 커리큘럼을 따라 커널 부팅·메모리 관리·프로세스·시스템콜·IPC·파일시스템까지 OS의 핵심 구성요소를 단계적으로 직접 구현했습니다. 특히 TLB Fast Refill, Copy-on-Write 기반 fork(), 사용자 공간 FS 서버와 IPC를 통한 마이크로커널 파일시스템 구조를 구현하며 운영체제 핵심 메커니즘을 깊이 실습했습니다.

주요 개발 내용 (Lab0 – Lab6)

  • 커널 부팅 및 메모리 초기화 (Lab0–1): GXemul 시뮬레이터 환경에서 부트스트랩 코드 작성, 물리 메모리 레이아웃 설정 및 커널 엔트리 포인트 초기화.
  • 가상 메모리 체계 구축 (Lab2): 2단계 페이지 테이블 기반 가상 주소 공간 설계, TLB Miss 시 Fast Refill 핸들러를 어셈블리로 직접 작성해 TLB 관리 오버헤드 최소화.
  • 프로세스 스케줄링 (Lab3): struct Env 기반 프로세스 관리 구조 설계, 라운드 로빈 스케줄러 구현, 컨텍스트 스위칭 시 레지스터 상태 저장·복원 로직 직접 작성.
  • 시스템콜 및 예외 처리 (Lab3–4): 시스템콜 인터페이스 설계, 사용자 모드·커널 모드 전환 시 Trapframe을 통한 레지스터 일괄 저장 및 복원 구현.
  • Copy-on-Write fork() 구현 (Lab4): 사용자 공간에서 COW 기반 fork()를 구현했습니다. 쓰기 가능한 페이지는 공유 후 TLB Mod 예외가 발생할 때만 실제 복사되도록 설계해 메모리 복사 비용을 지연시켰습니다.
  • Pipe IPC 메커니즘 (Lab6): 공유 페이지 기반 pipe와 파일 디스크립터 복제(dup)를 구현했습니다. pageref 기반 닫힘 판정과 map/unmap 순서를 함께 설계해 pipe 상태를 안정적으로 유지했습니다.
  • 사용자 공간 파일시스템 (Lab5): 커널 내부가 아닌 독립된 FS 서버 프로세스와 IPC로 통신하는 마이크로커널 구조로 파일시스템을 구현했습니다. 블록 캐시, 파일 열기·매핑·크기 조정·동기화 경로를 구성했습니다.

트러블슈팅

Pipe IPC 구현 중 매핑 순서에 따른 참조 수(pageref) 판정 오류 해결
Kernel / IPC
상황

dup() 및 close() 실행 시, FD와 Pipe 페이지의 매핑·해제 순서에 따라 pageref 값이 일시적으로 의도와 다르게 관측되는 레이스 컨디션이 발생했습니다.

대안 탐색

인터럽트 비활성화를 통한 임계 구간 보호를 검토했으나, 시스템 성능 저하와 교육용 OS의 설계 원칙을 고려해 제외했습니다. 대신 매핑 순서 재배치(Reordering)를 통해 항상 안전한 상태 불변식을 유지하는 방안을 선택했습니다.

해결

pageref(fd) ≤ pageref(pipe) 조건이 어떤 순간에도 깨지지 않도록 unmap·map 순서를 조정했습니다. IPC의 안정성을 확보했습니다.

pipe.c / fd.c — Invariant-Preserving map/unmap Order
// 해제 순서 조정으로 불변식 유지
static int pipe_close(struct Fd *fd) {
  // [Fix] FD를 먼저 해제하여
  // pageref(fd) > pageref(pipe)
  //   찰나의 순간을 방지
  syscall_mem_unmap(0, fd);
  syscall_mem_unmap(0, (void *)fd2data(fd));
  return 0;
}

선점형 스케줄링 환경에서는 '완성된 결과'뿐만 아니라 '과정 중의 중간 상태'도 안전해야 함을 깨달았습니다.
