Pipe IPC 구현 중 매핑 순서에 따른 참조 수(pageref) 판정 오류 해결

Kernel / IPC
상황

dup() 및 close() 실행 시, FD와 Pipe 페이지의 매핑·해제 순서에 따라 pageref 값이 일시적으로 의도와 다르게 관측되는 레이스 컨디션이 발생했습니다. 이로 인해 파이프 상대편이 닫힌 것으로 오인하여 통신이 비정상 종료되는 버그가 있었습니다.

대안 탐색

인터럽트 비활성화를 통한 임계 구간 보호를 검토했으나, 시스템 성능 저하와 교육용 OS의 설계 원칙을 고려해 제외했습니다. 대신 별도의 락 없이도 매핑 순서 재배치(Reordering)를 통해 항상 안전한 상태 불변식을 유지하는 최적화 방안을 선택했습니다.

해결

pageref(fd) ≤ pageref(pipe) 조건이 어떤 순간에도 깨지지 않도록 unmap·map 순서를 조정했습니다. 원자성이 보장되지 않는 구간에서도 참조 수를 기반으로 한 파이프 생존 판정이 항상 올바르게 작동하도록 설계하여 IPC의 안정성을 확보했습니다.

pipe.c — Atomic-like Mapping Sequence 
// 해제 순서 조정을 통해 불변식 유지
        void pipe_close(int fdnum) {
        // [Fix] FD를 먼저 해제하여 
        // pageref(fd)가 pipe보다 커지는 찰나의 순간을 방지
        syscall_mem_unmap(0, fd_to_va(fdnum));
        syscall_mem_unmap(0, pipe_to_va(fdnum));
        }
알게된 점

선점형 스케줄링 환경에서는 '완성된 결과'뿐만 아니라 '과정 중의 중간 상태'도 안전해야 함을 깨달았습니다. 코드 실행 순서만으로 동시성 문제를 해결하며, 방어적 설계와 불변식 유지의 중요성을 체감했습니다.

파서의 Fault-Tolerant 파싱 및 에러 복구 설계

Compiler
상황

세미콜론이 누락된 코드를 파싱할 때 동일한 오류가 중복 출력되는 문제가 발생했습니다. 토큰 스트림을 추적한 결과, 오류 복구 과정에서 토큰 포인터가 두 칸 전진하며 다음 위치에서 동일한 오류가 재발하는 구조적 결함을 확인했습니다.

대안 탐색

임시로 토큰 위치를 보정하는 방식과 파서 구조 자체를 수정하는 방식을 비교했습니다. 전자는 다른 구문에서도 같은 문제가 반복될 가능성이 있어, 근본적인 복구 로직 개선이 필요하다고 판단했습니다.

해결

Parser.cpp — Error Recovery 
void parseErrorPrintfI() {
        Token cur  = getCurrentToken();
        Token prev = getPreviousToken();

        if (cur.type == SEMICN) {
            outputToken(cur);
            advanceToken();
        } else {
            // 누락(i) 에러 - 직전 토큰 라인 기록
            // 파싱을 중단하지 않고 계속 진행
            addError(prev.lineNumber, 'i');
        }
        }

상황

ResponseResultHandler.java 
@RestControllerAdvice
        public class ResponseResultHandler
            implements ResponseBodyAdvice<Object> {

        @Override
        public Object beforeBodyWrite(Object body, ...) {
            // 이미 공통 래퍼면 그대로 반환
            if (body instanceof ResultWrapper)
            return body;
            // 데이터를 규약에 맞춰 자동 래핑
            return ResultWrapper.success(body);
        }
        }

초기 구조에서는 추론 지연과 모바일 환경의 자원 부담이 커서, 실시간 모니터링 서비스에 바로 적용하기 어려운 문제가 있었습니다.

① 해결 방안

모델 업데이트 유연성과 모바일 메모리 부담 완화를 위해 서버 사이드 추론 방식을 채택하고, 모바일은 백엔드 프록시를 통해 결과만 전달받도록 구조를 분리했습니다.

② 상황 — 저메모리 EC2의 OOM 장애

AWS 저사양 인스턴스에서 컨테이너 빌드 또는 AI 서비스 동시 구동 시 메모리 사용량이 급증하면서 운영 안정성이 떨어지는 문제가 있었습니다.

② 해결 — Pull-only 배포 전략 및 리소스 제한

로컬 또는 CI 환경에서 이미지를 사전 빌드해 레지스트리에 Push하고, 운영 서버는 Pull-only 전략으로 전환했습니다. 또한, mem_limit 기반 자원 제한과 AI 컨테이너의 profile 분리를 적용해 저메모리 환경에서도 배포 안정성을 확보했습니다.

docker-compose.ec2-lite.yml 
services:
        backend:
            image: ghcr.io/zzooonn/carelink/backend:latest
            # build 섹션 제거 (EC2 내 빌드 방지)
            mem_limit: 384m

        ai:
            image: ghcr.io/zzooonn/carelink/ai:latest
            profiles: ["ai"] # 필요 시에만 분리 실행
            mem_limit: 512m
알게된 점

제한된 인프라 자원에서는 단순 성능 튜닝보다, 아키텍처 분리·배포 전략·자원 제한을 함께 설계하는 접근이 전체 시스템 안정성에 더 직접적으로 기여한다는 점을 체감했습니다.
